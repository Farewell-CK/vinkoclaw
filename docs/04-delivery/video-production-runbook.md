# Video Production Runbook

## 输入

- 目标：产出 1 支项目介绍视频，时长建议 3-4 分钟。
- 约束：案例不能翻车，且必须有吸引力。
- 现状：系统已有审批、角色 skill、团队路由、审计与反思链路。

## 处理流程

### 1. 录制结构

- Part A（直播主线，约 2 分 30 秒）：按 `walkthrough-script.md` 主案例录制。
- Part B（预录高光，约 30 秒）：插入 frontend/backend 路由片段。
- Part C（结尾总结，约 20 秒）：强调 DGX Spark 本地团队操作系统定位。

### 2. 分镜脚本（可直接照读）

镜头 S1：定位（10 秒）

- 旁白：`这不是聊天机器人，这是运行在 DGX Spark 上的 AI 团队操作系统。`
- 画面：控制台首页 + 角色列表 + 任务队列。

镜头 S2：治理（35 秒）

- 指令：`请配置研究助理的记忆为向量数据库`
- 画面：审批单出现 -> 点击审批 -> 状态成功。
- 旁白：`关键配置必须审批，所有动作可追踪。`

镜头 S3：边界（30 秒）

- 指令：`给 ceo 安装向量记忆 skill`
- 画面：CEO skill 变化，其他角色不变。
- 旁白：`能力按角色精确分配，不做全员泛化。`

镜头 S4：团队执行（60 秒）

- 指令：`团队执行：生成3分钟功能介绍结构、核心卖点、风险清单`
- 画面：任务被多角色接力执行，最终产出 deliverable + reflection。
- 旁白：`不是一个 agent 回答问题，而是一个团队协同交付。`

镜头 S5：高光插片（20-30 秒）

- 插片内容：官网需求路由 Frontend、API 需求路由 Backend。
- 旁白：`系统会根据任务类型自动分派到最合适角色。`

镜头 S6：收尾（15 秒）

- 旁白：`飞书群是指挥室，DGX Spark 是 AI 总部。VinkoClaw 让一人团队具备企业级协同能力。`

### 3. 失败切换口令（必须准备）

如果 S4 响应变慢，立即切换到以下任一命令：

- `暂停模板 tpl-opc-internet-launch`
- `启用模板 tpl-opc-internet-launch`
- `切换开发模型到 glm-5`

切换话术：

- `这里重点是可治理能力，不依赖单次长推理的偶然表现。`

## 状态变化检查

录制中至少捕捉到 3 类状态变化：

- 审批状态变化（pending -> approved）
- 角色能力状态变化（某角色 skill 更新）
- 任务状态变化（queued -> running -> completed）

## 输出物

视频输出建议：

- 文件名：`vinkoclaw-project-intro-2026-04-09.mp4`
- 时长：180-240 秒
- 分辨率：1920x1080
- 码率：>= 8 Mbps

字幕输出建议：

- 保留关键术语字幕：`审批` `审计` `多角色协同` `DGX Spark` `reflection`

## 上下游影响

- 上游：依赖控制台可访问、至少一个可用通道、基础环境已启动。
- 下游：视频内容将直接作为“方案报告”和“技术文章”的叙事主线，减少重复改稿成本。

## 假设与未验证前提

- 假设：录制时控制台页面与任务执行服务处于可用状态。
- 假设：你可接受“主直播 + 预录高光”混合剪辑方案。
- 未验证前提：发布平台对视频大小、编码格式的限制需最终在提交通道确认。
