# Walkthrough Script (Project Introduction Video)

## 目标

这支视频只证明 3 件事：

1. VinkoClaw 是“AI 团队系统”，不是聊天机器人。
2. DGX Spark 在这里不是模型盒子，而是本地 AI 总部。
3. 系统具备可落地的治理闭环：审批、审计、角色边界、可观测执行。

## 案例策略（防翻车）

采用三层结构，兼顾稳定性和吸引力：

- 主案例（直播）：走高稳定路径，只用已验证能力。
- 高光案例（预录）：补充更完整的业务价值片段。
- 兜底案例（秒切）：主线任一环节异常时 30 秒内切换，不中断叙事。

## 主案例（直播，约 3 分钟）

案例名：`一人团队 3 分钟完成“能力配置 + 团队执行 + 结果复盘”`

### 镜头 1（0:00-0:25）开场定位

- 打开 DGX Spark 本机 + 控制台首页。
- 旁白：`飞书群是指挥室，DGX Spark 是 AI 总部。`
- 画面要点：角色列表、任务队列、审批入口。

### 镜头 2（0:25-1:00）审批与治理

- 输入命令：`请配置研究助理的记忆为向量数据库`
- 查看系统生成审批单 -> 点击审批 -> 状态变更成功。
- 讲解点：配置变更不是“悄悄改”，而是“可审批、可审计”。

### 镜头 3（1:00-1:35）角色边界（精准能力分配）

- 输入命令：`给 ceo 安装向量记忆 skill`
- 查看 CEO 的 skill 变化，其他角色不变。
- 讲解点：不是全员共享工具，而是岗位级能力治理。

### 镜头 4（1:35-2:35）团队执行（吸引力核心）

- 输入命令：`团队执行：生成3分钟功能介绍结构、核心卖点、风险清单`
- 查看自动路由与任务推进（例如 Product/Research/Engineering/QA 协同）。
- 查看最终输出包含：可执行方案 + 风险与改进建议（reflection）。

### 镜头 5（2:35-3:00）收束

- 查看任务结果页中的 summary、deliverable、reflection。
- 收束话术：
  - `这不是一个会聊天的 Agent。`
  - `这是一个可指挥、可审计、可复盘的 AI 团队。`

## 高光案例（预录，20-40 秒插入）

案例名：`产品官网任务路由到 Frontend，后端 API 任务路由到 Backend`

- 插入片段 A：产品官网类需求进入 Frontend 角色执行。
- 插入片段 B：后端 API 类需求进入 Backend 角色执行。
- 讲解点：不是“同一个大模型硬做所有事”，而是按角色分工执行。

用途：

- 作为主案例镜头 4 的增强证据。
- 如果直播执行时间不足，可用预录片段快速补全“工程能力”叙事。

## 兜底案例（任何时刻可切，约 45 秒）

如果团队执行耗时过长或模型响应波动，立刻切以下指令：

- `暂停模板 tpl-opc-internet-launch`
- `启用模板 tpl-opc-internet-launch`
- `切换开发模型到 glm-5`

兜底讲解点：

- 运营控制动作支持自然语言解析。
- 系统变更路径有审批与审计记录。
- 控制台可观测状态变化，不是黑盒调用。

## 录制前检查清单（必须）

- 已启动：`npm run dev`
- 控制台可打开：`http://127.0.0.1:8098`
- 通道状态已检查（飞书/邮件至少一个可用）
- 准备好 1 个审批动作、1 个 skill 变更、1 个团队执行任务
- 准备高光预录片段（防临时波动）

## 话术红线

- 不承诺“100% 自动写完全部生产代码”
- 不把“模型能力”讲成“产品能力”
- 重点讲“流程闭环和可治理”，突出工程成熟度

## 假设与未验证前提

- 假设：录制机器上模型服务可用，或可接受 fallback 输出完成流程验证。
- 假设：飞书/邮件至少一条通道具备可用状态。
- 未验证前提：网络质量与第三方服务稳定性不可控。
