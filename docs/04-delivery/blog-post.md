# VinkoClaw：在 DGX Spark 上运行一支 AI 团队

> 这不是聊天机器人，这是一个 AI 团队操作系统。

---

## 故事从一台 DGX Spark 开始

NVIDIA DGX Spark 发布的时候，很多人都在问：这台个人超级计算机能用来做什么？

有人说是本地推理盒子，有人说是实验平台。但我想，它应该是一个**AI 总部的物理载体**——一个让一个人拥有企业级 AI 团队的基础设施。

于是我做了一个项目：**VinkoClaw**。

它的目标很简单：**让一个创业者通过飞书群聊，就能指挥一支多角色 AI 团队在本地 DGX Spark 上完成工作。**

不是和一个 AI 助手聊天。而是指挥一个团队：产品经理写 PRD，前端工程师写页面，后端工程师写 API，测试工程师写用例，CEO 做决策，CTO 审架构。

---

## 为什么不用现成的 Agent 框架？

我试过很多。OpenClaw、AutoGen、CrewAI... 它们都很好，但有一个共同的问题：

**它们把 AI 做成了一个"超级个体"，而不是一个"团队"。**

当你需要写一个产品页面 + 后端 API + 测试用例时，单 Agent 框架会让你反复在上下文里切换角色。而 VinkoClaw 从一开始就设计成了多角色架构：

```
你 → 飞书群 → VinkoClaw 调度器
                   ↓
        意图分类 → 路由到合适的角色
                   ↓
        Product → PRD
        Frontend → 页面代码
        Backend → API 代码
        QA → 测试用例
                   ↓
        汇总 → 交付给你
```

## 现在能做什么？

### 1. 多角色 AI 团队

系统内置 **12 个专业角色**：

- **CEO** — 业务优先级、资源决策
- **CTO** — 技术架构、风险评估
- **产品经理** — PRD、需求拆解、验收标准
- **前端工程师** — React/Vue 组件、页面、样式
- **后端工程师** — API 设计、数据模型、服务
- **工程师** — 生产代码、脚本、bug 修复
- **开发者** — 通用脚本、工具、自动化
- **测试工程师** — 测试用例、回归策略
- **研究员** — 市场调研、竞品分析
- **算法工程师** — LLM 策略、模型优化
- **UI/UX** — 交互设计、视觉规范
- **运营** — 邮件、内容创作、文案

每个角色有独立的系统提示词和技能边界。

### 2. 智能路由

你说一句话，系统自动判断应该交给谁：

- `帮我写个用户登录功能的PRD` → 产品经理
- `帮我做一个登录页，用 React 写` → 前端工程师
- `帮我写一个后端 API，实现用户注册` → 后端工程师
- `我们系统的技术架构该如何设计` → CTO
- `帮我分析一下具身智能的市场现状` → 研究员
- `团队执行：做一个带登录和仪表盘的 SaaS MVP` → 多角色协作

路由是三层机制：模板匹配 → LLM 意图分类 → 关键词兜底。

### 3. 审批与安全治理

这是 VinkoClaw 和大多数 Agent 框架最大的区别：**它不是一个无政府系统。**

- 配置变更 → 需要审批
- 技能安装 → 需要审批
- 高风险代码执行 → 需要审批
- 邮件发送 → 需要审批

审批通过飞书交互式卡片进行，内联批准/拒绝按钮。所有决策写入审计日志。

低风险操作可以配置自动审批（由 CTO 角色自动批准），但依然保留审计记录。

### 4. 角色级技能绑定

在传统单 Agent 系统中，安装一个 skill（比如向量记忆）后，所有角色都能用。能力边界是模糊的。

VinkoClaw 的设计是：**技能精确绑定到具体角色。**

你在飞书里说"给 CEO 安装向量记忆 skill"，系统会：
1. 创建审批单
2. 审批通过后，仅 CEO 获得向量记忆能力
3. 产品经理、研究员等其他角色不受影响
4. 写入审计日志

这在 OPC（一人公司）场景下至关重要——不同岗位不应该拥有完全相同的能力。

### 5. 反思机制

每个任务完成后，系统自动生成一份反思报告：

- **质量评分** — 完成质量打分
- **置信度** — 对结果的信心程度
- **假设** — 执行中做了哪些假设
- **风险** — 识别到了哪些风险
- **改进建议** — 下次可以怎么做得更好

这让系统天然具备复盘能力。你不再需要猜 AI 对自己的结果有多少把握。

### 6. 本地推理 + 高可用

推理层设计为可插拔：

- 主推理：vLLM 运行 Qwen 3.5-35B-A3B FP8 量化版
- 兜底：SGLang / Ollama
- 降级：确定性本地输出（推理完全不可用时保证流程不中断）

即使模型服务暂时不可用，系统的审批链路、控制台、队列管理仍然正常工作。

### 7. 开发者工具执行链

Developer 角色可以通过 opencode 调用本地模型执行代码型任务：

1. 自动风险分级（检测 deploy / rm -rf 等关键词）
2. 低风险 CTO 自动批准，高风险 Owner 审批
3. 执行轨迹写入 `tool_runs`，控制台实时显示

### 8. Web 控制台

除了飞书，还有一个功能完整的 Web 控制台：

- 任务下发与管理
- 审批通过/拒绝
- 队列监控（深度、吞吐、SLA）
- 通道就绪状态（飞书/邮件配置完整度）
- 工具提供方可用性
- 审计事件查询
- 协作时间线

界面采用 Apple 风格设计，支持中英文切换。

## 技术实现

### 架构

```
TypeScript Monorepo
├── apps/          # 飞书网关 + Web 控制台
├── services/      # 调度器 + 任务执行器 + 邮件收件
├── packages/      # Agent 运行时 + 知识库 + 共享模块 + 插件 SDK
└── prompts/       # 12 个角色系统提示词
```

### 数据层

使用 Node.js 内置 SQLite（`node:sqlite`），无需额外数据库：
- 任务、审批、审计、技能绑定、协作记录、用户认证、工具执行记录

### 推理

通过 OpenAI 兼容 API 对接 vLLM/SGLang/Ollama，深度适配 Qwen 3.5 的 CoT 模式：
- 分类路径关闭 thinking（延迟敏感）
- 工具调用路径关闭 thinking（避免思考消耗工具 token）
- 纯完成路径开启 thinking + finalize 二次调用

## 未来规划：超级 OPC 操作系统

> **OPC = One-Person Company（一人公司）**

### 当前正在进行

- 真实飞书群联调
- 控制台 UI 全面优化（Apple 风格）
- 命令安全策略增强
- Provider 健康检查

### 下一阶段

- **更大上下文窗口** — 利用 DGX Spark 本地 GPU 内存
- **多角色并行推理** — 多 GPU 同时执行多个角色推理
- **本地向量库** — 长时记忆与 RAG
- **混合知识底座** — 文档/代码/邮件/日历统一检索
- **长任务持续运行** — 小时级任务的稳定执行

### 更远

- CRM 集成 — 客户线索管理
- 日历协同 — AI 自动排程
- 内容生产闭环 — 调研→写作→审核→发布
- 多项目工作台 — 同时管理多个产品线
- 插件市场 — 角色技能模板生态
- 可审计商业运营流 — 从线索到收入的完整审计链

## 快速开始

```bash
git clone git@github.com:Farewell-CK/vinkoclaw.git
cd vinkoclaw
npm install
cp config/.env.example .env
# 编辑 .env，设置模型后端
npm run dev
# 打开 http://127.0.0.1:8098
```

默认登录：`admin` / `vinkoclaw`
